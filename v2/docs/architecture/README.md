# v2 architecture

A high-level map of how v2 is shaped. For decision rationale, see `../adr/`. For day-to-day rules, see `../../CLAUDE.md`.

## One-paragraph summary

A pnpm monorepo with two deployable apps (`apps/web` React + Vite, `apps/api` Express) and four shared packages (`packages/db`, `packages/shared`, `packages/ui`, `packages/config`). Content (blog, ensayos, manifiesto, courses) lives as MDX in `content/`. The database is one Postgres instance on Neon. Auth uses httpOnly cookies + CSRF, bcrypt-hashed passwords, JWT access (15m) + refresh (30d) tokens.

## Request lifecycle (logged-in user)

1. Browser sends request with `credentials: 'include'`. Cookies travel automatically.
2. `securityHeaders()` (helmet) → `corsMiddleware()` → `requestId()` (UUID) → JSON body parser → `cookieParser` → general rate limiter.
3. `/api/auth/*` mounts before global CSRF — login/register can't yet have a CSRF cookie.
4. `csrfProtect()` runs for everything else that mutates state.
5. `authenticate()` reads `eihg_access` cookie → verifies JWT → sets `req.user` (typed via `Express.Request` augmentation).
6. Feature route → service → repository → Drizzle → Postgres.
7. Response goes back through the error handler (which redacts sensitive fields) and the request-id is exposed via `X-Request-Id`.

## Where to add things

| You want to add… | Put it here |
|---|---|
| A new HTTP endpoint | `apps/api/src/features/<domain>/routes.ts` |
| A new database table | `packages/db/src/schema/<domain>.ts`, then `pnpm --filter @v2/db db:generate` |
| A new repository | `packages/db/src/repositories/<domain>.ts`, export from index |
| A new client page | `apps/web/src/pages/<Page>.tsx`, lazy-load in `App.tsx` |
| A new shared component | `apps/web/src/components/...` (or `packages/ui` if cross-app) |
| A new env var | Add to `apps/api/src/lib/config.ts` schema, `env.example`, and `v2/.env` |
| Long-form content | `content/<kind>/<slug>.mdx` |

## Out of scope (deliberately deferred)

- Production deployment (Vercel project, DNS) — Phase 10
- httpOnly→localStorage data migration — fresh user base, no migration needed
- Server-side rendering — SSG only at build time for blog/ensayos/courses
- Real-time collaboration — there is no MVP requirement for it
